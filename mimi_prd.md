# PRD: mimi

## 1. 文档信息

- 项目名称：`mimi`
- 产品类型：macOS 桌面陪伴工具
- 当前阶段：MVP / 1.0
- 默认角色形象：短毛狸花猫
- 目标平台：macOS

## 2. 产品背景

在 LLM 使用场景中，用户输入 Prompt 后通常需要等待几秒到十几秒，期间容易出现注意力中断、等待焦虑和界面空窗感。`mimi` 的目标不是提升生产效率本身，而是在这段等待时间中提供轻量、克制、不打扰的情绪陪伴。

`mimi` 通过一个常驻桌面的宠物角色，将用户当前的输入焦点或鼠标位置转化为角色的视线与状态变化，成为工作流中的视觉锚点。

## 3. 产品目标

### 3.1 核心目标

1. 在用户等待 LLM 输出或短暂停顿时，提供轻量陪伴感。
2. 用最小打扰的方式融入桌面工作流，不影响用户正常点击、输入和切换窗口。
3. 用简单但自然的状态反馈，让角色具备“感知用户正在忙 / 正在等 / 已经停下”的生命感。

### 3.2 MVP 目标

1. 成功在 macOS 桌面渲染一个透明、置顶、无边框的宠物窗口。
2. 在支持的原生应用中获取文本光标位置，并驱动角色视线跟随。
3. 在无法获取文本光标的位置场景中，自动降级为跟随鼠标。
4. 根据用户操作节奏切换 `Focus`、`Wait`、`Idle` 三种基础状态。

### 3.3 非目标

以下内容不纳入 MVP：

- 多角色切换
- 宠物养成、互动喂养、任务系统
- 跨平台支持（Windows / Web）
- 复杂语音、聊天或 AI 对话能力
- 高复杂度骨骼动画系统

## 4. 目标用户

### 4.1 核心用户

- 高频使用 ChatGPT、Claude、Copilot、Cursor 等 LLM 工具的用户
- 长时间在 macOS 上进行写作、编程、研究、办公的知识工作者
- 对桌面陪伴感、情绪价值和轻量拟人反馈有偏好的用户

### 4.2 使用场景

1. 用户在 IDE、文档工具或聊天界面输入 Prompt 后等待回复。
2. 用户在多个窗口间切换，注意力容易漂移，希望有一个始终存在但不打扰的视觉陪伴物。
3. 用户短时间停顿思考，希望界面上有轻微反馈，而不是完全静止。

## 5. 用户问题

1. 等待 LLM 响应期间，界面反馈贫乏，容易出现“空转感”。
2. 用户注意力在短暂停顿中容易离散，缺少视觉锚点。
3. 传统桌面宠物往往抢交互、挡内容或资源占用过高，不适合现代工作流。

## 6. 产品方案概述

`mimi` 是一个默认常驻桌面的透明宠物窗口。它优先感知用户的文本输入光标位置，并根据位置调整眼睛、头部等视觉朝向；若当前应用无法提供文本光标信息，则自动切换到鼠标跟随模式。

同时，系统基于用户输入频率和停顿时长驱动角色状态机，在不同阶段展示不同动画：

- `Focus`：高频输入或快速移动
- `Wait`：短暂停顿，模拟等待和思考
- `Idle`：长时间无操作，进入休眠

整个应用默认不接受鼠标事件，不抢占焦点，不阻断底层操作；设置入口统一由菜单栏管理。

## 7. 功能需求

### 7.1 桌面宠物展示

**目标**
在桌面显示一个始终可见、透明、无边框的宠物角色。

**需求说明**

- 宠物窗口保持全局置顶
- 窗口背景完全透明
- 窗口默认无标题栏、无边框
- 窗口不主动抢占键盘焦点

**验收标准**

- 在主屏可稳定显示角色
- 切换应用时角色仍保持显示
- 宠物显示不会出现明显闪烁、黑底、窗口边框残留

### 7.2 事件穿透与无感共存

**目标**
保证宠物角色存在但不妨碍用户操作底层应用。

**需求说明**

- 默认开启鼠标事件穿透（Ignores Mouse Events）
- 用户点击宠物覆盖区域时，事件直接落到底层窗口
- 所有设置操作通过菜单栏图标进入，不在宠物主体上承载复杂交互

**验收标准**

- 宠物覆盖在编辑器、浏览器、聊天窗口之上时，不影响点击和输入
- 用户可通过菜单栏完成基础设置和退出操作

### 7.3 光标 / 鼠标追踪

**目标**
让角色尽量跟随用户的真实注意力焦点。

**需求说明**

- 优先获取当前文本输入光标位置
- 若获取失败，则回退为跟随系统鼠标位置
- 跟随结果应驱动角色眼睛、瞳孔或头部方向变化

**系统依赖**

- 鼠标追踪：`NSEvent.addGlobalMonitorForEvents`
- 光标追踪：`Accessibility API` + `AXUIElement`

**验收标准**

- 在备忘录、Xcode 等原生应用中可获取文本光标位置
- 在不支持或获取失败的场景中，可稳定切换到鼠标跟随
- 切换过程无明显卡顿或状态错误

### 7.4 状态感知与状态机

**目标**
基于用户操作频率表达角色的“忙碌 / 等待 / 休息”状态。

**状态定义**

1. `Focus`
   - 触发条件：连续键盘输入或鼠标快速移动
   - 表现：耳朵竖起，眼睛聚焦，头部做轻微跟随

2. `Wait`
   - 触发条件：无操作持续 3-10 秒
   - 表现：揣手、打哈欠、缓慢甩尾、视线放松

3. `Idle`
   - 触发条件：无操作超过 10 秒
   - 表现：闭眼休眠、身体轻微呼吸起伏、偶发耳朵抖动

**验收标准**

- 能根据输入节奏自动切换三种状态
- 状态切换自然，不出现高频抖动
- 每种状态至少有一种明确可见的动画或姿态反馈

### 7.5 权限引导

**目标**
确保用户理解并完成辅助功能授权，否则核心功能无法生效。

**需求说明**

- 首次启动时检测辅助功能权限状态
- 若未授权，展示清晰引导说明
- 引导用户前往系统设置中的“隐私与安全性 -> 辅助功能”
- 未授权时允许宠物基础显示，但需明确提示“文本光标追踪不可用”

**验收标准**

- 未授权时用户能明确知道缺失能力和处理路径
- 授权后无需复杂重试流程即可恢复核心能力，或有明确刷新提示

### 7.6 菜单栏管理

**目标**
为 MVP 提供最简配置入口和应用生命周期管理。

**需求说明**

- 应用在菜单栏提供图标
- 菜单栏至少包含以下操作：
  - 查看权限状态
  - 打开设置或帮助说明
  - 退出应用

**验收标准**

- 菜单栏图标稳定存在
- 用户可通过菜单栏完成退出和权限查看

## 8. 交互原则

1. 不打扰优先：角色存在感大于交互感，不主动索取用户操作。
2. 情绪价值优先：动画以陪伴和反馈为目标，而非夸张表演。
3. 克制优先：动作频率、幅度、切换节奏都应柔和，不制造噪音。
4. 可预期优先：角色行为应能被用户直觉理解，不出现随机惊扰。

## 9. 技术与实现约束

### 9.1 技术栈

- 使用原生 `Swift + AppKit`
- MVP 阶段不采用跨平台框架

### 9.2 系统能力依赖

- 依赖 macOS `Accessibility API` 获取文本光标
- 依赖全局事件监听获取鼠标和输入活动

### 9.3 坐标系统要求

- 必须支持多显示器环境
- 需要建立稳定的全局屏幕坐标到宠物窗口本地坐标的转换能力
- 眼球或视线方向计算需使用明确的角度计算方式，如 `atan2`

### 9.4 性能要求

- 常驻状态下 CPU 占用控制在 1-2% 以下
- 内存占用保持轻量
- 不出现明显掉帧、发热异常或电量负担过高

## 10. 兼容性与降级策略

### 10.1 支持场景

- macOS 原生文本输入应用，如备忘录、Xcode 等

### 10.2 风险场景

- 部分 Electron 或非标准输入控件应用，可能无法返回准确文本光标位置

### 10.3 降级策略

- 文本光标不可用时，自动切换为鼠标跟随
- 降级过程需连续、即时，不要求用户手动切换
- 产品层需允许“光标跟随不可用但应用仍可工作”的状态存在

## 11. MVP 验收标准

### 11.1 功能验收

1. 成功渲染无边框、透明、置顶的桌面宠物
2. 成功获取原生应用中的文本光标位置
3. 成功在不支持场景中回退为鼠标跟随
4. 成功切换 `Focus`、`Wait`、`Idle` 三种状态
5. 成功完成辅助功能权限引导

### 11.2 体验验收

1. 不拦截底层点击
2. 不主动抢焦点
3. 动作表达清晰但不过度打扰
4. 状态切换平滑自然

### 11.3 性能验收

1. CPU 占用满足目标范围
2. 常规使用下不造成明显卡顿和发热

## 12. 版本规划建议

### 12.1 MVP / 1.0

- 单角色狸花猫
- 光标 / 鼠标跟随
- 三态状态机
- 菜单栏入口
- 权限引导

### 12.2 后续可选方向

- 多角色皮肤
- 动作资源扩展
- 跟随位置和大小自定义
- 不同等待场景的动画包
- 与 LLM 客户端状态联动

## 13. 开发优先级建议

按以下顺序推进更稳妥：

1. 透明置顶窗口与事件穿透
2. 辅助功能权限检测与引导
3. 文本光标获取 PoC
4. 鼠标跟随 Fallback
5. 多屏坐标转换
6. 基础状态机
7. 动画表现接入
8. 菜单栏管理与打磨

## 14. 风险与待确认项

1. 不同应用对 `Accessibility API` 的支持差异较大，文本光标获取成功率需尽早验证。
2. 多屏坐标转换是核心技术风险，需要尽快通过 PoC 验证。
3. 动画资源如果过重，可能影响性能目标，需要控制实现复杂度。
4. `Wait` 与 `Idle` 的时间阈值后续可能需要通过真实使用体验再调整。
