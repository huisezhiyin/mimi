# Spec: mimi 注意力与跟随行为收敛

## 1. 背景

当前版本存在两个体验问题：

1. 用户输入时，宠物并不会稳定盯住文本输入焦点，而是经常退回鼠标。
2. 用户期待的角色感受不是“泛化跟随器”，而是：
   - 输入时像猫一样监督用户打字
   - 大模型生成时像猫一样好奇地看生成内容

## 2. 本次目标

本次只收敛注意力与表现，不扩展复杂动画系统。

目标：

1. 提高“文本输入阶段”对文本焦点的优先级和稳定性。
2. 给等待阶段增加明确的“好奇看输出”视觉语义。
3. 在等待阶段引入局部屏幕变化检测，作为“可能正在生成”的辅助信号。
4. 保持现有 `Focus / Wait / Idle` 状态机，不引入大范围架构改造。

非目标：

1. 不做真正的 LLM 输出内容语义理解。
2. 不做跨应用 DOM / WebView 语义解析。
3. 不做全屏持续智能分析。
4. 不做复杂骨骼动画或大规模角色系统重构。

## 3. 目标行为

### 3.1 输入监督态

当最近发生键盘输入时：

1. 优先使用文本光标作为注意力目标。
2. 若文本光标短暂读取失败，不立即切回鼠标，而是保留最近一次成功的文本焦点一小段时间。
3. 只有当持续无法读取文本焦点时，才退回鼠标。

用户体验应表现为：

- 猫在用户输入时更像盯着输入位置
- 不会因为一次 AX 读取失败就频繁左右跳
- 姿态上更像“认真监督”而不是“兴奋围观”

输入监督态的推荐形态：

1. 头部与视线更稳，优先锁定输入点
2. 眼睛略窄，眉线更明显，形成专注感
3. 嘴巴闭合，减少夸张表情
4. 尾巴只做轻微摆动或基本稳定
5. 默认不冒泡，避免对真实输入形成打扰

### 3.2 等待好奇态

当用户停止输入并进入 `Wait` 时：

1. 若近期存在成功的文本光标位置，优先继续看该位置附近。
2. 视线应比 `Focus` 更柔和，但要有“盯着屏幕内容等待结果”的感觉。
3. 不应在 `Wait` 态立刻被鼠标轻微移动打断成明显鼠标跟随。

这里的“看大模型生成”在 MVP 内采用近似策略：

- 若存在结构化生成信号（如 `Codex` provider），优先使用生成信号
- 否则不直接识别生成内容语义
- 以“最近一次文本输入焦点”为锚点
- 在该锚点附近做局部屏幕变化检测
- 若局部区域持续变化，则把该变化区域视为“可能生成中”

当存在结构化生成信号且阶段为 `preparing / streaming / settling` 时，角色表现应比普通 `Wait` 更主动：

1. `streaming`
   - 眼神更亮、更聚焦
   - 耳朵与尾巴更精神，形成“盯住生成”的感觉
   - 允许出现极短的自言自语气泡，例如“唔...”
2. `settling`
   - 保持短暂的“看结果”状态
   - 表情比 `streaming` 更放松
   - 允许出现短促确认气泡，例如“嗯？”
3. `preparing`
   - 作为轻量预备态，只做很弱的期待感，不做夸张演出

等待好奇态的推荐形态：

1. 核心感受应是“好奇围观”，而不是“输入监督”的延续
2. `streaming` 时头部可略前倾，耳朵更立、更朝前
3. 眼睛更亮、更圆一点，但不能做成惊吓表情
4. 嘴巴允许轻微张开，尾巴更有活力
5. 自言自语气泡必须低频、极短，只用于点缀，不得持续打扰

### 3.3 休息态

当进入 `Idle` 时：

1. 弱化主动跟随
2. 允许瞳孔回中或轻微下垂

### 3.4 同时事件的优先级

当多个事件同时发生时，统一按以下顺序仲裁：

1. 显式生成信号中的 `streaming`
2. 显式生成信号中的 `preparing`
3. 显式生成信号中的 `settling`
4. 用户正在输入
5. 最近输入位置保活
6. 局部屏幕变化
7. 鼠标
8. `idle`

解释：

1. 只要存在可靠的大模型生成信号，优先看生成，不再被输入态或鼠标轻易打断
2. 没有可靠生成信号时，才退回“监督输入”
3. 再次降级时，才使用局部屏幕变化与鼠标

对于多个同时活跃的生成会话，宠物始终只跟随一个主会话，不做“小分身”。

主会话选择优先级：

1. 前台窗口对应的会话优先
2. `streaming` 高于 `preparing`
3. `preparing` 高于 `settling`
4. 最近更新时间更新的会话优先
5. 若仍相近，优先保持当前已选主会话，避免频繁抖动

## 4. 最小实现方案

涉及文件：

1. `mimi/App/AppDelegate.swift`
2. `mimi/Tracking/TrackingTarget.swift`
3. `mimi/Character/PetView.swift`
4. `mimi/Support/ScreenChangeMonitorService.swift`
5. `mimi/MenuBar/MenuBarController.swift`

最小实现切口：

1. 在跟随链路里增加“最近一次成功文本目标”的缓存与保活时间。
2. 新增轻量注意力模式：
   - `typing`
   - `waitingForResponse`
   - `idle`
3. `typing` 时优先文本光标，并对短暂失败做粘性保留。
4. `waitingForResponse` 时优先看最近文本目标，而不是立刻退回鼠标。
5. 基于最近文本焦点附近的局部截图做轻量 diff，提取变化热点。
6. `PetView` 根据模式调整瞳孔偏移强度和眼型，让“监督”与“好奇”有差别。
7. 当 `Codex` provider 提供 `GenerationPhase` 时，`PetView` 额外根据 `streaming / settling / preparing` 区分表情、尾巴与短气泡。
8. 表现层与跟随层解耦：
   - 跟随层回答“看哪里”
   - 表现层回答“现在像监督、围观还是休息”

## 5. 风险

1. 当前没有真实“LLM 正在生成”的系统级信号，只能通过局部屏幕变化做代理。
2. 局部变化可能来自光标闪烁、页面动画或其他 UI 刷新，存在误判。
3. 某些应用长期无法返回文本光标时，仍然只能退回鼠标。
4. 若未授予屏幕录制权限，等待态只能退回“盯最近文本焦点”的近似行为。
5. 如果瞳孔偏移幅度过大，可能从“监督”变成“惊吓”。

## 6. 验证

1. 连续输入时，猫应主要盯文本输入区域，而不是频繁跳回鼠标。
2. 停止输入 3 到 10 秒后，若附近区域持续发生屏幕变化，猫应转而盯变化热点，形成“等回复”的感觉。
3. 若没有屏幕变化或没有屏幕录制权限，猫应继续看最近输入区域，而不是立刻跳鼠标。
4. 在不支持文本光标的应用中，仍可退回鼠标，但切换不应抖动。

## 7. Change Log

截至 `2026-03-20`，已完成以下实现落地：

1. 在 `AppDelegate` 中加入最近文本光标缓存、输入态保活、等待态目标选择与局部屏幕变化观察接入。
2. 在 `TrackingTarget` 中新增文本光标保活与局部屏幕变化两类目标来源。
3. 新增 `ScreenCapturePermissionService`，提供屏幕录制权限检测、请求与设置跳转。
4. 新增 `ScreenChangeMonitorService`，围绕最近文本焦点附近区域做轻量截图 diff 并提取变化热点。
5. 在菜单栏中加入屏幕录制权限状态与操作入口。
6. 在 `PetView` 中加入 `AttentionMode` 表现差异，让监督输入、好奇等待、休息中三种眼神更容易区分。
7. 在 `PetView` 中加入基于 `GenerationPhase` 的轻量活跃演出，让 `Codex` 生成中的 `streaming / settling` 不再只表现为文字徽标变化。
8. 明确了输入监督态、生成好奇态与多事件同时发生时的统一优先级。

## 8. Validation

1. 当前只完成静态实现与文档同步，未运行 Xcode 编译。
2. 局部屏幕变化检测的真实效果仍需用户在本机验证，尤其是：
   - ChatGPT / Claude / Cursor 等 LLM 界面
   - 已授予与未授予屏幕录制权限两种场景
   - 输入区在底部、输出区在上方的聊天式布局
