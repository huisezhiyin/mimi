# Spec: mimi MVP 开发拆解

## 1. 文档定位

本文档用于把 [mimi_prd.md](/Users/wuyue/github_project/mimi/mimi_prd.md) 转成可执行的 MVP 开发规格与 checkpoint 清单，作为后续设计、实现、验收的直接依据。

约束原则：

- `Spec is Truth`
- 代码改动前先基于本 spec 收敛最小改动集
- 按 checkpoint 分阶段推进，不跨阶段扩散需求
- 每阶段先完成最小可验证链路，再补细节

## 2. 本轮目标

MVP 只解决以下问题：

1. 在 macOS 桌面渲染一个透明、置顶、事件穿透的宠物窗口
2. 获取用户关注点，优先文本光标，失败时回退到鼠标
3. 通过最小状态机表达 `Focus`、`Wait`、`Idle`
4. 提供最小菜单栏管理与辅助功能权限引导

本轮不做：

- 多角色系统
- 素材编辑器
- AI 对话能力
- 复杂互动系统
- 高复杂度动画引擎

## 3. 总体开发顺序

建议严格按以下顺序推进：

1. 工程骨架
2. 透明窗口与事件穿透
3. 权限检测与引导
4. 文本光标获取 PoC
5. 鼠标跟随 fallback
6. 坐标转换
7. 基础状态机
8. 菜单栏管理
9. 角色表现接入
10. 验收与性能收敛

原因：

- `文本光标获取` 和 `坐标转换` 是技术风险最高的链路，必须尽早验证
- `窗口层`、`权限层`、`跟随层` 是所有表现层能力的前置依赖
- 动画表现应晚于技术链路打通，否则容易过早投入在错误方向

## 4. Checkpoint 拆解

### CP0: 文档冻结

**目标**
冻结当前 MVP 范围，避免边做边扩。

**输入**

- [mimi_requirement.md](/Users/wuyue/github_project/mimi/mimi_requirement.md)
- [mimi_prd.md](/Users/wuyue/github_project/mimi/mimi_prd.md)

**输出**

- 本 spec 作为后续研发依据

**完成标准**

- MVP 范围、非目标、开发顺序明确
- 团队对“先验证底层链路，再接表现层”达成一致

### CP1: 工程骨架

**目标**
建立可运行的原生 macOS AppKit 工程骨架。

**范围**

- 创建原生 Swift + AppKit 项目
- 明确应用入口、窗口控制器、菜单栏入口、基础目录结构
- 预留追踪服务、状态机、渲染层的模块边界

**建议模块**

- `App`：应用入口与生命周期
- `MenuBar`：菜单栏与基础操作
- `Window`：透明置顶宠物窗口
- `Tracking`：光标 / 鼠标获取
- `State`：状态机
- `Character`：角色渲染与动作表现
- `Support`：权限、坐标转换、公共工具

**完成标准**

- 工程可启动
- 主窗口与菜单栏入口已建立
- 模块目录和职责清晰，未引入多余抽象

### CP2: 透明窗口与事件穿透

**目标**
先打通“能显示且不打扰”的最小链路。

**范围**

- 无边框窗口
- 透明背景
- 全局置顶
- 不抢焦点
- 鼠标事件穿透

**关键决策**

- 宠物窗口只承载显示，不承载复杂设置交互
- 设置统一从菜单栏进入

**完成标准**

- 角色占位图或占位视图可稳定显示
- 用户点击宠物区域时，底层窗口可正常响应
- 切换应用时宠物仍维持可见

### CP3: 权限检测与引导

**目标**
确保文本光标能力具备前置条件。

**范围**

- 检测辅助功能权限状态
- 未授权提示说明
- 提供前往系统设置的引导
- 明确无权限时的降级状态

**完成标准**

- 启动时可判断权限状态
- 未授权时用户知道缺了什么能力
- 未授权状态下应用仍可继续运行基础展示

### CP4: 文本光标获取 PoC

**目标**
验证 `Accessibility API` 是否能拿到原生应用中的文本光标位置。

**范围**

- 获取当前焦点元素
- 获取选区或插入点范围
- 查询文本光标 bounds
- 输出全局坐标

**验证场景**

- 备忘录
- Xcode
- 其他至少一个原生文本输入场景

**完成标准**

- 在至少两个原生应用中成功获取文本光标坐标
- 能识别获取失败场景并返回明确失败结果

**风险**

- 不同应用对辅助功能属性支持不一致
- 可能出现焦点元素可读但光标 bounds 不可读的情况

### CP5: 鼠标跟随 Fallback

**目标**
在文本光标不可用时，保证跟随能力不失效。

**范围**

- 接入全局鼠标监听
- 构建统一的“当前目标点”抽象
- 光标优先，鼠标兜底

**建议抽象**

- `TrackingTarget.cursor`
- `TrackingTarget.mouse`
- `TrackingTarget.unavailable`

**完成标准**

- 光标可用时优先使用光标
- 光标不可用时自动退化为鼠标
- 切换过程对上层渲染透明

### CP6: 坐标转换

**目标**
把系统全局坐标稳定映射到宠物窗口本地空间。

**范围**

- 支持多显示器
- 处理窗口位置与屏幕坐标差异
- 输出可用于视线朝向计算的本地目标点

**完成标准**

- 在单屏和多屏场景下都能正确映射
- 角色朝向不会出现明显反向、偏移或镜像错误

### CP7: 基础状态机

**目标**
建立不依赖复杂动画系统的最小行为表达。

**状态**

- `Focus`
- `Wait`
- `Idle`

**输入信号**

- 键盘活动频率
- 鼠标移动速度或活跃度
- 无操作持续时间

**建议规则**

- 高频输入或快速移动进入 `Focus`
- 3 到 10 秒无操作进入 `Wait`
- 超过 10 秒进入 `Idle`
- 任何新的明显活动都应打断 `Wait` / `Idle` 并返回 `Focus`

**完成标准**

- 三态切换可观察
- 无明显来回抖动
- 阈值集中配置，便于后调

### CP8: 菜单栏管理

**目标**
提供 MVP 所需的最小操作入口。

**范围**

- 菜单栏图标
- 权限状态查看
- 帮助说明或设置入口
- 退出应用

**完成标准**

- 应用可通过菜单栏完成基础管理
- 不需要依赖宠物本体点击交互

### CP9: 角色表现接入

**目标**
在前述能力稳定后，再接入最小视觉表现。

**范围**

- 先用占位角色视图或简单资源接入
- 先实现眼睛 / 头部方向变化
- 再补 `Wait` / `Idle` 的轻量动作

**完成标准**

- 用户可明显感知“正在跟随”和“状态已切换”
- 视觉表现不要求复杂，但必须稳定且克制

### CP10: 验收与性能收敛

**目标**
按 PRD 验收，并压住常驻资源开销。

**验收维度**

- 功能
- 体验
- 权限
- 降级
- 性能

**完成标准**

- 通过 PRD 中的 MVP 核心验收项
- 常驻 CPU 维持在目标范围附近
- 无明显高频刷新导致的风扇、发热或掉帧问题

## 5. 研发执行规则

后续实现阶段，默认遵守以下规则：

1. 每次代码改动先说明本次只覆盖哪个 checkpoint。
2. 每次只做一个最小闭环，不同时推进多个高风险点。
3. 先用占位实现打通链路，再补 UI 和表现细节。
4. 影响需求、行为、接口或约束时，必须同步更新 spec 或 PRD。
5. 如果底层能力验证失败，先回写风险与替代方案，不直接硬做表现层。

## 6. 建议的首个实现批次

如果下一步进入代码阶段，建议第一批只做：

1. `CP1 工程骨架`
2. `CP2 透明窗口与事件穿透`
3. `CP3 权限检测与引导`

原因：

- 这三项构成最小可见、可运行、可继续验证的应用外壳
- 它们对后续 `Accessibility API` PoC 是必要前置
- 范围可控，适合作为第一轮代码改动的最小方案

## 7. 待确认项

进入代码前，仍建议明确以下口径：

1. 第一版角色表现是使用静态位图、矢量绘制，还是纯占位圆形视图
2. 第一版菜单栏是否需要“开机启动”选项
3. 第一版是否只支持主屏显示，还是立即支持跨屏宠物定位
4. 第一版权限引导是只提示说明，还是要直接跳系统设置

## 8. 当前实现口径（2026-03-20）

以下内容已按 `CP1` 到 `CP10` 的当前范围收口：

1. 工程形态为原生 `Swift + AppKit`，使用本地 Xcode 工程骨架组织源码。
2. 应用当前以菜单栏工具形态运行，`Info.plist` 中启用 `LSUIElement`，不依赖 Dock 图标作为主入口。
3. 宠物窗口当前为主屏右下区域的固定占位窗口，尺寸 `220 x 220`，用于先验证透明、置顶、事件穿透链路。
4. 角色表现当前使用代码绘制的占位狸花猫视图，后续可替换为正式素材或更细的渲染实现。
5. 权限引导当前包含三条路径：启动提示、菜单栏主动请求权限、菜单栏直接打开辅助功能设置。
6. 已接入一个 CP4 级别的文本光标获取 PoC，可通过菜单栏主动读取当前焦点文本光标并输出全局坐标或明确失败类型。
7. 已接入全局鼠标监听和统一 `TrackingTarget` 抽象，当前可通过菜单栏主动验证“文本光标优先，失败后回退鼠标”的链路。
8. 已接入全局到宠物窗口本地空间的坐标映射能力，可通过菜单栏主动验证本地目标点，并在占位视图中绘制映射点。
9. 已接入最小活动信号监听与 `Focus / Wait / Idle` 状态机，当前会按定时刷新更新占位视图状态，并可通过菜单栏主动读取判定结果。
10. 已接入最小连续跟随链路，当前会以轻量定时刷新持续更新跟随目标与本地坐标，占位角色会根据本地目标点调整瞳孔朝向。
11. 已补充 [mimi_acceptance_checklist.md](/Users/wuyue/github_project/mimi/mimi_acceptance_checklist.md) 作为 `CP10` 的验收执行文档，覆盖手动验证步骤、性能观察建议和残留风险。
12. 当前阶段尚未执行编译、真机运行、性能采样和最终人工验收；`CP10` 目前完成的是验收材料准备，不是运行验证完成。
13. 当前阶段尚未接入更复杂的头部跟随、尾巴动作和持续视线动画逻辑，这些仍归后续 checkpoint 处理。
